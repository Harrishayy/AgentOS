# P3 merge plan — CLI + manifest from `P3`

**Source:** `origin/P3`
**Target:** `integration/p1-p2-p3` (branched from `integration/p1-p2`)

## What we want from `P3`

- `cmd/agentctl/` — the cobra CLI entrypoint
- `internal/cli/`, `internal/manifest/`, `internal/daemon/`, `internal/version/`
- `e2e/` — testscript end-to-end tests
- `examples/` — manifest examples (will replace any Mehul-era stubs)
- `go.mod`, `go.sum` (root module: `github.com/agent-sandbox/cli`)
- `project-plan.md`, `ASSUMPTIONS.md`
- `planning/p3/...`, `plans/p3-*.md`
- `.github/workflows/ci.yml` (extends Harrish's)

## Multi-module concern

The repo will end up with two Go modules:
- `./go.mod` — `github.com/agent-sandbox/cli` (P3, the CLI)
- `./daemon/go.mod` — daemon module (P2)

This is fine. To make `go vet ./...` from the root sane, we add a
`go.work` file at root with both modules. Without it, `go vet ./...` from
root will skip `daemon/` (because it's a nested module) which is acceptable.

We will create a `go.work` to make the workspace explicit.

## Conflicts expected

- `examples/` — Mehul's stubs (`allowed-net.yaml`, etc.) vs P3's
  (`web-fetcher.yaml`, etc.). Keep both, but if there is any name collision
  prefer P3's content (it is schema-validated).
- `.github/workflows/ci.yml` — concatenate Harrish's daemon job with P3's
  CLI job (two separate jobs in the same workflow file). Inspect both before
  resolving.
- `setup-vm.sh` — already canonical from P2 step; do not pull P3's version.
- `Vagrantfile` / `README.md` / `CONTRIBUTING.md` — keep the integration
  versions (which match `main`).

## Steps

1. `git checkout -b integration/p1-p2-p3 integration/p1-p2`
2. Pull P3 territory:
   ```
   git checkout origin/P3 -- cmd/ internal/ e2e/ go.mod go.sum \
     project-plan.md ASSUMPTIONS.md planning/ plans/p3-week3.md plans/p3-week3-build.md plans/p3-gap-review.md
   git checkout origin/P3 -- examples/web-fetcher.yaml examples/file-reader.yaml examples/llm-agent.yaml examples/shell-runner.yaml examples/bad-cidr.yaml examples/bad-paths.yaml
   ```
3. Resolve `.github/workflows/ci.yml`: read both, combine into one workflow
   with `daemon-ci` and `cli-ci` jobs. (If they look largely independent,
   we can ship a single multi-job file.)
4. Create `go.work`:
   ```
   go 1.23
   use ./
   use ./daemon
   ```
5. Commit: `integrate: P3 — agentctl CLI, manifest schema, e2e tests`

## Smoke test

```bash
go vet ./cmd/... ./internal/...     # repo-root module
go build -o /tmp/agentctl ./cmd/agentctl
/tmp/agentctl --help
/tmp/agentctl manifest validate examples/web-fetcher.yaml
go test ./internal/manifest/...     # unit tests
```

If `go.sum` requires fetching deps and the sandbox has no network, the build
will fail. In that case record the failure mode and continue (the missing
deps will resolve when CI runs).
